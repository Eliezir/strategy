Alguém já escreveu — ou leu — algo assim?

public class Processador {

    public void executar(String feature, Request req) {
        switch (feature) {
            case "feature1":
                validar(req); salvar(req); notificar(req);
                break;
            case "feature2":
                validar(req); gerarRelatorio(req);
                break;
            case "feature3":
                aplicarDesconto(req); salvar(req);
                break;
            case "feature4": // ...
                break;
            case "feature5": // ...
                break;
            // ... e mais 10 cases ...
            default:
                throw new IllegalArgumentException("Feature desconhecida: " + feature);
        }
    }
}

Nem precisa explicar o que faz. O que você está sentindo já é a resposta.

Bloco 0 · Por que isto acontece

Ninguém escreve assim de propósito

Início
Projeto raramente nasce com escopo fechado. Você começa com 3 casos bem definidos.
Meio do caminho
Nem o próprio cliente sabe o que quer. Ele usa, pede ajustes, aparece feature nova. Depois outra. E outra.
6 meses depois
Aquele switch de 3 virou 15. Cada feature nova exige editar a mesma classe. Mudar uma coisa quebra outra.

"Isto não é incompetência — é a natureza do negócio de software. Padrões de projeto existem exatamente porque essa dor é inevitável."

Bloco 1 · Fundamentos

O que são padrões de projeto

"Soluções reutilizáveis e nomeadas para problemas que aparecem repetidamente no design de sistemas orientados a objetos."

Um padrão é a receita — não o bolo. Guia, não obriga.

Criacionais
Como objetos são criados.
Ex.: Singleton, Factory
Estruturais
Como objetos se compõem.
Ex.: Adapter, Decorator
Comportamentais
Como objetos colaboram.
Ex.: Observer, Strategy, State

Gang of Four (Gamma, Helm, Johnson, Vlissides) · 1994 · 23 padrões clássicos

5
Bloco 1 · Desmistificando

O que um padrão não é

  • Não é biblioteca. Você não import Strategy de lugar nenhum — você escreve no seu código.
  • Não é framework. Independente de Spring, Quarkus, qualquer tecnologia.
  • Não é obrigação. Usar padrão em código que não precisa é over-engineering.
  • É vocabulário compartilhado. Quando você diz "aqui é Strategy", quem conhece entende em 3 segundos o que você fez — e, mais importante, por quê.

Bloco 2 · Colaboração humana

Por que padrões importam hoje

Em qualquer projeto real, você passa mais tempo lendo código do que escrevendo.

Código é comunicação
Padrão é um protocolo entre devs que nunca se viram. Reduz discussão em code review e acelera onboarding.
Analogia — peças de xadrez
Em vez de "aquela peça que anda em L", você diz "cavalo". Um nome comum torna a conversa imediata.

Até aqui é o argumento clássico. Em 2026 existe um motivo novo — e é o mais interessante.

Bloco 3 · Strategy · Intenção

O padrão Strategy

"Define uma família de algoritmos, encapsula cada um e os torna intercambiáveis. Strategy permite que o algoritmo varie independentemente dos clientes que o usam."

Gamma, Helm, Johnson & Vlissides · Design Patterns (1994)

O problema
Objeto que se comporta de formas diferentes → vira switch gigante, acoplado, impossível de testar isolado. Viola o Open/Closed Principle.
A solução
Cada algoritmo numa classe própria, atrás de uma interface comum. O Context escolhe — e pode trocar em runtime — sem saber qual implementação está usando.
11
Bloco 3 · Estrutura

A estrutura do padrão

Context Carrinho
«interface» Strategy
concreteStrategy A
concreteStrategy B
concreteStrategy C
  • Context — guarda referência a uma Strategy. Delega a execução sem conhecer a implementação concreta.
  • Strategy (interface) — contrato único (1 ou poucos métodos).
  • ConcreteStrategy — cada algoritmo em sua própria classe.
  • Pra trocar em runtime: passar outra ConcreteStrategy. Sem recompilar o Context.
12
Bloco 3 · Essência

Strategy é polimorfismo com nome e intenção

Muita gente aprende polimorfismo em POO e depois aprende Strategy, e pensa "mas isso é só polimorfismo". É — mas com uma intenção nomeada.

O valor do padrão é comunicar: quando você diz FreteStrategy, qualquer leitor — humano ou agente — sabe imediatamente por que o polimorfismo está sendo usado ali.

Bloco 4 · Demo Java

Cenário — cálculo de frete em e-commerce

Um carrinho de compras precisa calcular o frete conforme a transportadora escolhida. Três opções iniciais:

Correios
peso × 2,5 + 10,00
Jadlog
peso × 3,0 + 5,00
Retirada
grátis

E se amanhã o chefe pedir "adicione Loggi"? Vamos ver como cada abordagem responde.

14
Bloco 4 · Antes

Sem Strategy — o if/else hell

anti-pattern 
public class CalculadoraFrete {

    public double calcular(String tipo, double pesoKg) {
        if (tipo.equals("CORREIOS")) {
            return pesoKg * 2.5 + 10.0;
        } else if (tipo.equals("JADLOG")) {
            return pesoKg * 3.0 + 5.0;
        } else if (tipo.equals("RETIRADA")) {
            return 0.0;
        }
        throw new IllegalArgumentException("Tipo inválido");
    }
}
Problemas
  • Strings mágicas — "correios" em minúsculo quebra.
  • Nova transportadora = editar esta classe. Viola OCP.
  • Testar Jadlog exige instanciar a calculadora inteira.
  • Cresce pra sempre: 3 hoje, 15 em 2 anos.
15
Bloco 4 · Depois · Passo 1 de 4

A interface — o contrato

strategy 
public interface FreteStrategy {
    double calcular(double pesoKg);
}
  • Um único método — define o contrato que toda estratégia de frete precisa implementar.
  • É o ponto de desacoplamento entre quem usa o frete e quem calcula.
  • Tão simples que Java a trata como functional interface — volta mais tarde.
16
Bloco 4 · Depois · Passo 2 de 4

Três estratégias, três classes

concrete 
public class FreteCorreios
        implements FreteStrategy {

    @Override
    public double calcular(
            double pesoKg) {
        return pesoKg * 2.5
             + 10.0;
    }
}
concrete 
public class FreteJadlog
        implements FreteStrategy {

    @Override
    public double calcular(
            double pesoKg) {
        return pesoKg * 3.0
             + 5.0;
    }
}
concrete 
public class FreteRetirada
        implements FreteStrategy {

    @Override
    public double calcular(
            double pesoKg) {
        return 0.0;
    }
}

Cada classe é testável isolada: new FreteJadlog().calcular(2.0) → 11.0. Nenhuma conhece as outras.

17
Bloco 4 · Depois · Passo 3 de 4

O Context — quem usa a estratégia

context 
public class Carrinho {

    private FreteStrategy frete;

    public Carrinho(FreteStrategy frete) {
        this.frete = frete;
    }

    public void setFrete(FreteStrategy frete) {
        this.frete = frete;
    }

    public double totalComFrete(double subtotal, double pesoKg) {
        return subtotal + frete.calcular(pesoKg);
    }
}
  • Carrinho só conhece a interface — nunca as classes concretas.
  • O setFrete permite trocar em runtime — ex.: usuário muda o CEP, sistema troca a estratégia sem recriar o carrinho.
  • Isso é injeção de dependência na veia, sem framework.
18
Bloco 4 · Depois · Passo 4 de 4

Troca em runtime

main 
public class App {
    public static void main(String[] args) {
        Carrinho carrinho = new Carrinho(new FreteCorreios());
        System.out.println(carrinho.totalComFrete(100.0, 2.0));

        carrinho.setFrete(new FreteJadlog());
        System.out.println(carrinho.totalComFrete(100.0, 2.0));

        carrinho.setFrete(new FreteRetirada());
        System.out.println(carrinho.totalComFrete(100.0, 2.0));
    }
}
output
$ java App 115.0 // Correios (2kg) 111.0 // Jadlog (2kg) 100.0 // Retirada (grátis)

Adicionar Loggi amanhã? FreteLoggi implements FreteStrategy. Zero alteração no Carrinho.

Bloco 4 · Bônus · se sobrar tempo

Java moderno — lambda como Strategy

java 8+ 
// FreteStrategy tem 1 método só
// → é uma functional interface

FreteStrategy freteGratis =
    peso -> 0.0;

FreteStrategy fretePromocional =
    peso -> peso * 1.0;

carrinho.setFrete(freteGratis);

Padrão não envelhece — a linguagem evolui pra expressá-lo melhor.

Estratégias pequenas, de escopo local, podem virar uma linha. O padrão continua lá — só mudou a sintaxe.

Bloco 5 · Fechamento

Strategy em três frases

  • 01
    O que é
    Strategy encapsula variações de um algoritmo em classes separadas, trocáveis em tempo de execução.
  • 02
    Quando usar
    Quando você tem várias formas de fazer a mesma coisa e elas tendem a crescer ao longo do tempo.
  • 03
    Por que importa agora
    Código padronizado é mais legível pra humanos e pra agentes de IA — e padrões persistem mesmo quando frameworks mudam.

Bloco 5 · Armadilhas

Quando não usar Strategy

Over-engineering
2 casos que nunca vão virar 5? Um if/else simples é mais honesto que interface + 2 classes.
Abstração vazia
Se todas as "estratégias" fazem quase a mesma coisa, você não está usando Strategy — está usando interface pra constar.
Dogma
Padrão é decisão de design, não obrigação. O padrão serve o projeto — não o contrário.

Use padrões pra problemas que doem. Código simples que nunca cresce é ok ser simples.

Bônus · Reserva

Strategy vs State — mesma estrutura, perguntas diferentes

Strategy
O cliente escolhe
  • Estratégias geralmente não se conhecem.
  • Quem troca é o cliente ou uma Factory externa.
  • Ex.: frete — o usuário escolhe a transportadora.
Pergunta que responde
"Qual algoritmo eu uso agora?"
State
O próprio objeto transita
  • Estados conhecem uns aos outros (sabem a próxima transição).
  • Quem troca é o próprio objeto, reagindo a eventos.
  • Ex.: Pedido: Pendente → Pago → Enviado → Entregue.
Pergunta que responde
"Em que momento do ciclo de vida estou?"
